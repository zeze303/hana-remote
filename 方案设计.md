# hana-remote 技术方案

> 通过 Web 浏览器远程使用 Hanako 的工作能力。
> 架构：浏览器 → Render 中继 ←outbound WSS← 工作电脑 Hanako 插件

---

## 一、整体架构

```
┌─────────────────────────┐      WSS       ┌─────────────────────┐      outbound WSS      ┌────────────────────┐
│  浏览器 (手机/平板/PC)   │ ─────────────→ │   Render 中继服务    │ ←──────────────────── │  工作电脑          │
│                         │                │  (Node.js)          │                       │  Hanako 插件       │
│  ┌───────────────────┐  │                │                     │                       │  ┌──────────────┐  │
│  │ 登录页             │  │                │  ┌───────────────┐  │                       │  │ ws-client.js  │  │
│  │  ↓登录成功后       │  │                │  │ 连接管理器     │  │                       │  │  ↓             │  │
│  │ ┌────────┬──────┐ │  │                │  │  Worker 池     │  │                       │  │ handlers/     │  │
│  │ │文件树  │ 标签页│ │  │                │  │  Client 池     │  │                       │  │ ├─ chat.js    │  │
│  │ │        │       │ │  │                │  │  (配对路由)     │  │                       │  │ └─ files.js   │  │
│  │ │        ├───────┤ │  │                │  └───────────────┘  │                       │  │           ↓      │  │
│  │ │        │聊天   │ │  │                │  ┌───────────────┐  │                       │  │    ┌──────────┐ │  │
│  │ │        │ 面板  │ │  │                │  │ 认证模块      │  │                       │  │    │ Hanako   │ │  │
│  │ │        │       │ │  │                │  │ (jwt + session)│  │                       │  │    │ 对话流    │ │  │
│  │ │        ├───────┤ │  │                │  └───────────────┘  │                       │  │    │ + 工具链  │ │  │
│  │ │        │编辑器 │ │  │                │  ┌───────────────┐  │                       │  │    └──────────┘ │  │
│  │ │        │(Monaco)│ │  │                │  │ 消息路由      │  │                       │  └────────────────┘  │
│  │ └────────┴──────┘ │  │                │  │ type: chat/    │  │                       └────────────────────┘
│  └───────────────────┘  │                │  │ file_ls/file_  │  │
└─────────────────────────┘                │  │ read/file_     │  │
                                           │  │ write/…        │  │
                                           │  └───────────────┘  │
                                           └─────────────────────┘
```

**关键设计：所有连接都由工作电脑向 Render 发起（outbound），不需要工作电脑有公网 IP。**

## 二、消息协议

所有通信走 JSON over WSS。消息格式统一：

```jsonc
// 请求
{
  "id": "msg_001",           // 唯一 ID，客户端生成
  "type": "chat | file_ls | file_read | file_write | file_stat | file_tree",
  "payload": { … }
}

// 成功响应
{
  "id": "msg_001",           // 对应请求 ID
  "ok": true,
  "type": "chat | file_ls | …",
  "payload": { … }
}

// 错误响应
{
  "id": "msg_001",
  "ok": false,
  "error": "错误描述"
}

// 推送消息（无需请求）
{
  "id": "push_001",
  "push": true,
  "type": "worker_connected | worker_disconnected | connection_status",
  "payload": { … }
}
```

### 消息类型

| type | 方向 | payload 说明 |
|------|------|-------------|
| `chat` | client → worker | `{ text: "用户消息" }` |
| `chat` | worker → client | `{ text: "回复内容", done: true/false }`（流式可多次） |
| `chat_cancel` | client → relay | 无 payload，relay 转发 cancel 信号给 worker |
| `file_tree` | client → worker | `{ path: "C:\\" }` |
| `file_tree` | worker → client | `{ path: "C:\\", children: [ { name, type: dir/file, size, mtime }, … ] }` |
| `file_read` | client → worker | `{ path: "C:\\file.txt" }` |
| `file_read` | worker → client | `{ path, content, size, mtime }` |
| `file_write` | client → worker | `{ path, content }` |
| `file_write` | worker → client | `{ path, ok: true }` |
| `file_stat` | client → worker | `{ path }` → `{ path, type, size, mtime, isBinary }` |
| `worker_connected` | relay → client | `{ connected: true }` |
| `worker_disconnected` | relay → client | `{ connected: false }` |

## 三、模块详细设计

### 3.1 Render 中继服务 (`relay/`)

**职责：**
- 接受浏览器发起的 WSS 连接（带 Token 认证）
- 接受工作电脑插件发起的 WSS 连接（带 Secret 认证）
- 将浏览器的请求路由到对应的工作端
- 将工作端的回复/推送转发回浏览器
- 托管前端静态文件
- 管理连接状态

**核心状态：**

```
┌──── ConnectionManager ────┐
│  worker: WebSocket | null  │  最多 1 个工作连接
│  clients: Map<id, WS>     │  多个浏览器连接
│  msgQueue: Map<id, WS>    │  待响应的请求 msgId → 目标 client
│  status: 'online'/'offline'│
└────────────────────────────┘
```

**消息路由逻辑：**
- client 发 `{ id, type, payload }` → relay 存 `msgQueue.set(id, client)` → 转发给 worker
- worker 回 `{ id, ok, payload }` → relay 查 `msgQueue.get(id)` → 转发回对应 client → 清理
- push 类消息 → relay 广播给所有 client

**认证：**

两个级别的认证：
- **Client 认证**：浏览器登录获取 JWT Token，WSS 握手时携带
- **Worker 认证**：内置 Secret，连接时验证

**部署配置（环境变量）：**

```
PORT=3000
JWT_SECRET=xxx
WORKER_SECRET=xxx
SESSION_TIMEOUT=1800000    # 30 分钟
RATE_LIMIT_LOGIN=5         # 5 次尝试
RATE_LIMIT_WINDOW=900000   # 15 分钟
```

### 3.2 Hanako 插件 (`plugin/`)

**职责：**
- 作为 full-access 插件加载到 Hanako
- 启动时主动连接 Render 中继
- 处理 `chat` 类型消息 → 注入 Hanako 对话流
- 处理 `file_*` 类型消息 → 调用文件系统操作后返回结果
- 断线自动重连

**连接管理：**

```javascript
// ws-client.js 核心逻辑
connect()
  → ws = new WebSocket(RELAY_URL)
  → 发送 auth: { type: "auth", role: "worker", secret: WORKER_SECRET }
  → 成功后开始收发消息
  → 断线时 setInterval(reconnect, 5000)
```

**Chat 处理（关键路径）：**

```javascript
// handlers/chat.js
async function handleChat(payload) {
  // payload: { text: "用户消息" }
  // 通过 Hanako 的工具系统注入消息
  // Hanako 的回复通过观察者模式捕获
  // 以流式 chunk 逐步推回 relay → 浏览器
}
```

这里有一个设计选择：插件是通过 API 向 Hanako 发送消息，还是通过进程间通信。具体要看 Hanako 插件 API 的能力：
- 如果 Hanako 的 full-access 插件能直接调用对话引擎 → 直接注入
- 如果只能通过文件/消息队列 → 写入一个临时会话，监听输出

**File 处理：**

```javascript
// handlers/files.js
// 直接用 Node.js fs 模块操作文件系统
// 注意路径在 Windows 下的处理（盘符、反斜杠）

async function handleFileTree(payload) {
  // payload.path = "C:\\"
  // 读取目录 → 列出所有文件和子目录
  // 标记文件类型（文本/二进制/文件夹）
  // 返回排序后的列表
}

async function handleFileRead(payload) {
  // 读文件内容
  // 如果文件 > 2MB 或为二进制 → 拒绝
}

async function handleFileWrite(payload) {
  // 写入前备份原文件到 .hana-remote-backups/
  // 写入内容
  // 返回结果
}
```

**安全：**
- 所有写入操作记录日志（谁、什么时候、改了哪个文件）
- 备份保留 7 天

### 3.3 Web 前端 (`web/`)

**整体布局：**

```
┌────────────────────────────────────────────────┐
│  [logo] hana-remote    [模式: 浏览/编辑] [⚪]   │  ← 顶部栏
├──────────────────┬─────────────────────────────┤
│  📁 此电脑       │  ┌─ 标签栏 ──────────────┐  │
│  ├─ C: (系统)    │  │ 聊天 | 文件: xxx.js  │  │
│  │ ├─ Program…   │  ├───────────────────────┤  │
│  │ ├─ Users      │  │                       │  │
│  │ └─ Windows    │  │  内容区域              │  │
│  ├─ D: (数据)    │  │  (聊天面板 / Monaco    │  │
│  ├─ E: (Work)    │  │   编辑器 / 文件预览)   │  │
│  │ ├─ code       │  │                       │  │
│  │ └─ ...        │  │                       │  │
│  └─ …            │  │                       │  │
│                   │  └───────────────────────┘  │
├──────────────────┴─────────────────────────────┤
│  状态栏: 连接状态 | 当前路径 | 文件大小         │  ← 底部栏
└────────────────────────────────────────────────┘
```

**页面：**

- **登录页** (`/`): 密码输入框 → POST → 获取 JWT → 跳转主页面
- **主页面** (`/app`): 双栏布局，通过 WSS 连接

**组件层级：**

```
App
├── AuthProvider（JWT 管理，localStorage）
├── WSProvider（单例 WSS 连接，自动重连）
├── ConnectionBar（顶部连接指示器 + 模式切换）
├── MainLayout
│   ├── FileTree
│   │   ├── DriveNode（盘符级）
│   │   ├── DirNode（目录级，可折叠）
│   │   └── FileNode（文件级，点击打开）
│   └── ContentArea
│       ├── TabBar（切换已打开的文件/聊天）
│       └── ContentPanel
│           ├── ChatPanel
│           │   ├── MessageList（流式渲染 markdown）
│           │   └── InputBar（发送/取消）
│           ├── FileEditor
│           │   ├── Monaco Editor（只读/编辑模式）
│           │   └── SaveButton（编辑模式才显示）
│           └── FilePreview
│               ├── ImagePreview（图片内嵌）
│               └── BinaryInfo（二进制仅信息）
└── Toast（通知组件）
```

**文件树懒加载：**
- 第一层只显示盘符（调用 `file_tree path=""`）
- 点击展开目录时调用 `file_tree path="C:\\Users"` 获取子级
- 展开状态存 localStorage，刷新后恢复

**聊天流式渲染：**
- 用户发送 → 消息立即出现在列表中（灰色）
- Hanako 回复以 chunk 形式到达 → 逐字追加到消息气泡
- 收到 `done: true` → 停止追加

**连接状态：**
- 顶部右侧圆点：绿色(已连接工作电脑) / 黄色(重连中) / 红色(离线)
- 离线时输入框禁用，显示提示文字

## 五、关键技术决策

### 为什么插件走 outbound WSS 而非 inbound

工作电脑可能没有公网 IP，也可能是内网。插件主动连接 Render 是最稳健的方案——只需要工作电脑能访问外网就行。Render 也不需要配负载均衡。

### 为什么不用 WebSocket 直连工作电脑

即使有公网 IP，也面临 IP 变动、防火墙、NAT 穿透等问题。通过 Render 中转，双方都只做 outbound 连接，是最干净的模式。

### 文件写入备份策略

每次保存文件，插件自动在文件同目录的 `.hana-remote-backups/` 下保留上一版本。命名格式：`文件名_YYYYMMDD_HHmmss.bak`。保留最近 7 天的备份。

### 文本文件识别

插件通过扩展名（.txt/.md/.js/.ts/.json/.html/.css/.py/.java/.c/.h/.xml/.yaml/.yml/.env/.gitignore/…）和 MIME 类型首字节判断。不认识的默认按二进制处理，不预览。

### 前端无框架

前端用原生 JS + CSS，不引入 React/Vue。原因：
- 页面复杂度可控，原生足够
- 减少打包体积
- 静态文件直接托管，没有构建步骤也能跑（Monaco 通过 CDN 引入）

## 七、部署流程

### Render 中继

1. 在 Render 创建 Web Service
2. Build Command: `npm install`
3. Start Command: `node index.js`
4. 绑定域名，自动 HTTPS
5. 设置环境变量（JWT_SECRET、WORKER_SECRET）

### 工作电脑插件

1. 把 `plugin/` 文件夹放入 Hanako 插件目录
2. 在 Hanako 设置中启用插件
3. 配置环境变量 `.env` 中的 `RELAY_URL` 和 `WORKER_SECRET`

### 前端

- 开发期：`npx serve web/` 本地调试
- 生产期：构建后放入 `relay/public/`，由 relay 统一托管
